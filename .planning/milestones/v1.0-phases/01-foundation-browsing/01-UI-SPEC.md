## Spacing Scale

Streamlit does not expose a CSS spacing API directly. The scale below governs:
- `gap` and `padding` in any `st.columns()` layouts
- `st.container()` sectioning choices
- Widget ordering decisions (what appears "together" vs "separated")

Declared values (multiples of 4px — expressed as logical units):

| Token | Value | Usage in Streamlit context |
|-------|-------|----------------------------|
| xs | 4px | Icon-text gap in inline badge text; internal widget micro-spacing |
| sm | 8px | Spacing between sibling widgets in sidebar (Streamlit inserts ~8px naturally) |
| md | 16px | Default vertical gap between logical groups (achieved via blank `st.write("")` or `st.divider()`) |
| lg | 24px | Gap between major sidebar sections (DB selectors vs Platform picker); use `st.divider()` |
| xl | 32px | Gap between tab content regions (pivot grid vs metric row above it) |
| 2xl | 48px | Major section breaks on Settings page (DB Connections section vs LLM Connections section) |
| 3xl | 64px | Not used in Phase 1 (no full-page hero layouts) |

Exceptions:
- Touch targets: not applicable (intranet desktop tool, mouse interaction only)
- `st.dataframe` internal cell padding: controlled by Streamlit/browser defaults — do not attempt to override

## Typography

Streamlit renders all text through its built-in heading/body hierarchy. The following roles map to Streamlit's native text primitives:

| Role | Streamlit Primitive | Size | Weight | Line Height |
|------|---------------------|------|--------|-------------|
| Caption | `st.caption()` | 12px | 400 | 1.4 |
| Body / Label | `st.write()` / widget `label=` | 14px | 400 body / 600 label | 1.5 |
| Heading | `st.subheader()` | ~20px | 600 | 1.2 |
| Display | `st.title()` / `st.header()` | ~28px | 600 | 1.1 |

<!-- ASSUMED: exact pixel sizes follow Streamlit's default Markdown-based typographic scale;
     no custom CSS injection required for Phase 1 -->

Rules:
- Use exactly 2 weight levels across all screens: regular (400) for caption and body, semibold (600) for labels, headings, and display. Do NOT use bold (700) anywhere.
- Size differential alone creates sufficient hierarchy between Heading (~20px) and Display (~28px); weight distinction is unnecessary at those sizes.
- `st.title()` appears once per page at most — do not use it for section headings.
- `st.caption()` (12px, weight 400) is reserved for: row-count indicator ("showing N of M platforms"), column-cap warning, export filename hint. Do not use `st.caption()` for widget labels or functional copy.
- Do NOT use `st.markdown("## ...")` as a substitute for `st.subheader()` — keep primitives consistent for the checker.

## Color

Streamlit theme colors drive the 60/30/10 split:

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#ffffff` (backgroundColor) | Main panel background, tab content area, dialog background |
| Secondary (30%) | `#f0f2f6` (secondaryBackgroundColor) | Sidebar background, Settings form card-like regions, `st.expander` backgrounds, `st.info` banner backgrounds |
| Accent (10%) | `#1f77b4` (primaryColor) | Exactly: primary action buttons (`st.button` with `type="primary"`), active tab indicator, Plotly first series color, hyperlinks |
| Destructive | `#d62728` (Plotly red / `st.error` red) | Used only in: connection test fail badge (inline ❌ text styled via `st.error()`), Settings page delete-entry confirm dialog warning text |

Accent reserved for:
1. The single "primary" call-to-action button per page/tab (Apply Filters on Browse, Save Connection on Settings, Download in export dialog)
2. Active tab indicator (Streamlit renders this automatically from `primaryColor`)
3. Plotly chart first data series fill/stroke
4. URL-copy success flash (one-time `st.success` toast)

Accent is NOT used for: general text, icons, dividers, disabled states, warning banners, row-count indicators.

<!-- ASSUMED: no brand color specified by user; #1f77b4 is the Streamlit/Plotly conventional blue.
     Destructive color #d62728 matches Plotly's built-in second categorical color — familiar to data users. -->

## Browse Page Contract

**Primary focal point:** pivot grid (`st.dataframe`) in the Pivot tab, occupying the full main panel.

### Tab Structure (from D-02, D-13)

Three tabs rendered with `st.tabs(["Pivot", "Detail", "Chart"])`. All tabs read from the same
`st.session_state` filter keys.

### Pivot Tab

**Above-grid controls row** (use `st.columns([1, 1, 4])` or similar narrow-left layout):
- Col 1: "Swap axes" toggle — `st.toggle("Swap axes", key="pivot_swap_axes")`
- Col 2: Row-count indicator — `st.caption("N platforms × K parameters")` <!-- source: BROWSE-06 -->
- Col 3: Export button — `st.button("Export...", type="secondary")` → triggers `st.dialog`

**Pivot grid:**
- `st.dataframe(df, use_container_width=True, hide_index=False)`
- Column config: apply `st.column_config.TextColumn` for all Result columns (values are heterogeneous — do NOT apply NumberColumn globally)
- `PLATFORM_ID` index column is always visible (frozen left column behavior in Streamlit)
- 30-column cap: when selected parameters exceed 30, truncate and show `st.warning("Showing first 30 of N parameters. Narrow your selection to see all.")`

**Empty state** (no platforms or parameters selected):
- `st.info("Select platforms and parameters in the sidebar to build the pivot grid.")`

**Loading state:**
- `st.spinner("Fetching data...")` wrapping the `ufs_service` call

**Error state:**
- `st.error("Could not load data. Check your database connection in Settings. ({short error})")`

**Apply Filters CTA:** Streamlit reruns on widget change (implicit apply). If an explicit button is rendered for any reason, the label MUST be `"Apply Filters"` (not `"Apply"`).

### Detail Tab

Shown only when exactly one platform is selected. Otherwise:
- `st.info("Select exactly one platform in the sidebar to see its full parameter detail.")`

**Layout:** `st.dataframe` of long-form `(InfoCategory, Item, Result)` rows, sorted by
`(InfoCategory ASC, Item ASC)`. No pivot — full parameter list for the platform.

Row-count indicator: `st.caption("K parameters across N categories")`

### Chart Tab (from D-13, D-14)

Controls (render before chart):
```
col_picker = st.selectbox("Column to chart", options=[numeric_cols_only])
chart_type = st.radio("Chart type", options=["Bar", "Line", "Scatter"], horizontal=True)
```

Chart: `st.plotly_chart(fig, use_container_width=True)`

**Empty state** (no numeric columns available): `st.info("No numeric columns in the current selection. Add numeric parameters in the sidebar.")`

**Empty state** (no selection yet): `st.info("Select platforms and parameters in the sidebar first.")`

## Settings Page Contract

**Primary focal point:** first expanded DB connection form.

### Layout (from D-09 to D-12)

Single-column layout. Two collapsible `st.expander` sections:
1. `st.expander("Database Connections", expanded=True)`
2. `st.expander("LLM Connections", expanded=True)`

(Collapsed on subsequent visits via `expanded=False` default — first load shows expanded.)
<!-- ASSUMED: single-column is sufficient; D-12 discretion note says "single vs two-column" is at
     Claude's discretion. Single column chosen for simplicity and mobile intranet compatibility. -->

### Per-Entry Form Pattern (DB and LLM)

Each connection entry renders as:
```
st.subheader("{entry name}")           ← e.g. "prod_db" or "ollama_local"
[form fields in st.columns([3, 1])]
  Left col (3): st.text_input fields for name, host, port, user, database
                st.text_input("Password / API Key", type="password") for credentials
  Right col (1): st.button("Test", key="test_{entry_name}")
                 [pass/fail badge rendered below Test button after click]
[st.button("Save Connection", type="primary")]    ← one Save Connection per entry (not per section)
[st.button("Delete", type="secondary")]
```

Pass/fail badge (from D-10):
- Pass: `st.success("Connected")` — inline below Test button
- Fail: `st.error("Failed")` + `st.expander("Error detail")` containing the raw exception message

Add new entry: `st.button("+ Add Database", type="secondary")` / `st.button("+ Add LLM", type="secondary")`
→ Appends a blank entry form below existing entries in the same expander.

**AgentConfig display** (from D-12 discretion):
- Render `AgentConfig` fields as read-only `st.text_input(disabled=True)` at the bottom of the LLM section under `st.caption("Agent defaults (Phase 2)")`.

**Save behavior** (from D-12):
- On successful save: `st.toast("Saved. Caches refreshed.")` (Streamlit `st.toast` is the correct primitive — dismisses automatically)
- Cache clear: `st.cache_resource.clear()` then `st.cache_data.clear()` immediately after `save_settings()`.

**Delete confirmation:**
- `st.dialog` titled "Delete connection?" with body "This will remove {name} from settings.yaml. This cannot be undone." and two buttons: `st.button("Delete", type="primary")` (red — use `st.button` with `type="primary"` inside the dialog; this is the only place a "primary" style button triggers a destructive action) and `st.button("Keep Connection", type="secondary")`.
<!-- NOTE: Streamlit does not natively support a "danger" button variant in 1.56.0;
     use st.button(type="primary") inside the delete dialog where the primary action IS destructive,
     and rely on the dialog title + body copy to communicate the destructive intent clearly.
     "Keep Connection" (not "Cancel") tells the user what NOT clicking Delete preserves. -->

## Export Dialog Contract (from D-16)

Triggered by `st.button("Export...")` above each tab. Rendered via `@st.dialog("Export data")`.

Dialog contents:
```
Format:   st.radio("Format", ["Excel (.xlsx)", "CSV (.csv)"], horizontal=True)
Filename: st.text_input("Filename", value="pbm2_{tab}_{ISO-timestamp}")
          st.caption("File will be saved to your Downloads folder.")
[Pivot tab only]
Scope:    st.radio("Scope", ["Current view (pivot)", "Raw long-form rows"], index=0)

st.button("Download", type="primary")  → triggers st.download_button flow
st.button("Close", type="secondary")   → closes dialog (non-destructive; no confirmation needed)
```

Filename sanitization: strip characters outside `[A-Za-z0-9_\-.]`; collapse repeated underscores;
max 128 chars. Apply silently without error message (just sanitize). <!-- ASSUMED -->

## Shareable URL Contract (from BROWSE-09)

Filter state keys written to `st.query_params`:
- `platforms`: comma-separated PLATFORM_ID values, URL-encoded
- `params`: comma-separated `InfoCategory/Item` strings, URL-encoded
- `swap`: `"1"` if axes are swapped, omitted otherwise

On page load: read `st.query_params`, populate `st.session_state` before widgets render.

URL-copy UI: `st.button("Copy link")` → writes URL to clipboard via
`st.components.v1.html('<script>navigator.clipboard.writeText(...)</script>', height=0)`.
On success: `st.toast("Link copied to clipboard.")`.

## Interaction States Contract

Every data-loading surface must implement all four states:

| State | Primitive | Trigger |
|-------|-----------|---------|
| Loading | `st.spinner("Fetching data...")` | While `ufs_service` call is in-flight |
| Empty | `st.info(...)` | Zero rows returned, or no selection made |
| Error | `st.error(...)` | Exception caught from DB adapter or service layer |
| Success | `st.dataframe(...)` or `st.plotly_chart(...)` | Data returned and pivot/chart ready |

No "skeleton" loaders, no custom progress bars. `st.spinner` is sufficient for the expected
query latency on a pre-filtered 100k-row EAV table (target: < 2s with server-side WHERE).

## Accessibility Contract

<!-- ASSUMED: intranet desktop-only tool; WCAG AA compliance is a best-effort target, not a hard requirement -->

- All `st.text_input`, `st.selectbox`, `st.multiselect`, `st.toggle` calls MUST include a non-empty `label=` parameter. Never pass `label_visibility="hidden"` unless a visible alternative label is present elsewhere on the screen.
- `st.dataframe` column names must be human-readable strings (not raw DB column identifiers). `PLATFORM_ID` is acceptable; `InfoCategory` should be relabeled to `Category` in display.
- `st.plotly_chart` figures must include `fig.update_layout(title=...)` with the parameter name as the chart title.
- Color is never the sole indicator of state: pass/fail badges use both color AND text ("Connected" / "Connection failed") AND icon (✓ / ✗ via `st.success`/`st.error` primitives which add icons automatically).
