<objective>
Close gap-2 from 04-HUMAN-UAT.md (severity: major) — Apply button does not swap pivot grid; only Swap-axes triggers render.

Diagnosed root cause (full evidence in `.planning/debug/gap-2-apply-no-swap.md`, high confidence): the Apply button in `_picker_popover.html` has `hx-include="#browse-filter-form input:checked"` which is a CSS descendant selector. Because the picker checkboxes live inside `.dropdown-menu` subtrees and not as DOM descendants of the (intentionally empty, sibling) `<form id="browse-filter-form">`, the selector matches zero elements. The Apply POST therefore arrives at `/browse/grid` with empty `platforms=[]` and `params=[]`, and `build_view_model.is_empty_selection` returns the empty-state alert.

The Swap-axes checkbox in `_filter_bar.html` works because it carries `form="browse-filter-form"` on the triggering element itself — HTMX's `dn()` (getInputValues) for non-GET requests calls `Nt(triggeringElement) = element.form`, and `form.elements` enumerates ALL form-associated controls (including those linked via `form=` even when not DOM descendants).

Purpose: A one-attribute fix that makes Apply use the same HTMX form-association path as Swap-axes — eliminating the divergence that caused gap-2.

Output:
- Single template attribute change in `app_v2/templates/browse/_picker_popover.html` (add `form="browse-filter-form"`, drop the broken `hx-include`)
- One end-to-end behavioral regression test (Apply submits checked items → grid renders, not empty state)
- One static smoke test on rendered popover HTML (`form="browse-filter-form"` is present on the Apply button) so a future macro refactor cannot silently re-open the gap
- gap-2 status update in `04-HUMAN-UAT.md` (open → resolved) with `fix:` and `resolved:` fields populated

Zero changes to `routers/browse.py`, `services/browse_service.py`, `services/ufs_service.py`, `adapters/db/mysql.py`, or any other Python module. The route handler is already correct — proven by Swap-axes working against the same view-model path.

        """gap-2 regression: the form body that HTMX produces post-fix renders the grid.

        Post-fix, when the user clicks Apply, HTMX's dn() function:
          1. resolves Nt(applyButton) = applyButton.form (= the empty
             #browse-filter-form, courtesy of the new form= attribute)
          2. fn() iterates form.elements (which includes the form-associated
             checkboxes inside the dropdown menus via the form= attr they
             ALREADY have on the <input type="checkbox"> elements)
          3. emits a POST body like:
                 platforms=P1&platforms=P2&params=attribute%20%C2%B7%20vendor_id

        This test sends exactly that body and asserts:
          - response is 200
          - response contains the populated pivot table (NOT the empty-state alert)
          - the platforms+params arrived at fetch_cells as a non-empty tuple
            (proves the form-association actually carried the values, not that
            the route handler accidentally tolerated the empty body).

        Before the fix the same Apply click sent platforms=&params= (or rather,
        no keys at all) → is_empty_selection=True → empty-state alert.
        """
        captured: dict = {}

        def _capture_fetch(db, p, ic, i, row_cap=200, db_name=""):
            captured["platforms"] = p
            captured["infocategories"] = ic
            captured["items"] = i
            return (
                pd.DataFrame({
                    "PLATFORM_ID": ["P1", "P2"],
                    "InfoCategory": ["attribute", "attribute"],
                    "Item": ["vendor_id", "vendor_id"],
                    "Result": ["0xA1", "0xB2"],
                }),
                False,
            )

        _patch_cache(
            monkeypatch,
            platforms=["P1", "P2"],
            params=[{"InfoCategory": "attribute", "Item": "vendor_id"}],
            fetch=_capture_fetch,
        )

        r = _post_form_pairs(client, "/browse/grid", [
            ("platforms", "P1"),

            ("params", "attribute · vendor_id"),
