## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Tampering | `requirements.txt` (BS4 + lxml supply chain) | accept | Project-wide pin style is lower-bound + major-cap; no hash pinning anywhere in the file. Out of scope to introduce pip-tools / hash pins now. BS4 and lxml are widely-used libraries; risk on PyPI is low for this internal intranet tool. |
| T-01-02 | Information Disclosure | Jinja2 autoescape on `entity_id` (template var) | mitigate | Every `{{ entity_id }}` reference uses the explicit `| e` filter (matches existing `{{ platform_id | e }}` pattern at the same call sites). Default Jinja2 autoescape is also ON for `.html` templates — belt-and-suspenders. |
| T-01-03 | Information Disclosure | Jinja2 autoescape on `summary_url` (URL string in `hx-post`) | mitigate | `summary_url` is server-controlled (constructed via f-string from a router-validated path; in JV plan, the path arg is regex-bound to `^\d+$`). Still escape with `| e` to defend against any future route that takes user-influenced input. The `| e` filter neutralizes `"` and `>` which would otherwise break out of the attribute. |
| T-01-04 | Elevation of Privilege | `summary_html | safe` (LLM output rendering) | accept | Phase 3 invariant: LLM Markdown output is rendered through `MarkdownIt('js-default')` server-side BEFORE reaching the template, which sanitizes HTML to a safe subset. The `| safe` here only marks the already-sanitized HTML as not-double-escape. Out of scope to revisit. |
