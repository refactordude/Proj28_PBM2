# Phase 3 — UI Design Contract

> Visual and interaction contract for the Ask-feature overhaul into a multi-step agentic chat. Generated by `gsd-ui-researcher`, anchored to Dashboard_v2.html palette already pulled into v2.0 via Phase 02. **Most decisions are already locked in `03-CONTEXT.md` (15 D-CHAT-* IDs); this UI-SPEC pins the gap items (exact px values, selector names, truncation cap, motion, focus, and copywriting) into a prescriptive contract for the planner and executor.**

## Spacing Scale

The existing 4-multiple scale already in use across v2.0 carries forward verbatim. **No new tokens.**

| Token | Value | Usage on Ask page |
|-------|-------|-------------------|
| xs | 4px | Thought-pill vertical inner padding (`4px 12px`); `.chat-thought` border-left thickness |
| sm | 8px | Thought-pill block margin (`margin: 8px 0`); Stop button vertical padding |
| md | 16px | Final-card chart top margin (18px is acceptable Phase 4 deviation, see exceptions) |
| lg | 24px | (existing) detail-page section padding |
| xl | 32px | Final-card horizontal panel-body padding (matches existing `.panel-body { padding: 26px 32px }`) |
| 2xl | 48px | (unused on Ask) |
| 3xl | 64px | (unused on Ask) |

**Exceptions (codified, not drift):**
- `.panel-body` carries `26px 32px` (Phase 02 token-locked verbatim from Dashboard_v2.html). The `26px` vertical is a Dashboard_v2 verbatim port. **Do not change to 24px** — it is matched by `.panel-header { 18px 26px }` and the symmetry across panel header / body / footer is intentional.
- `.chat-pill-tool-call` uses `padding: 6px 12px` (research-recommended) — 6px is OFF-scale by 2px but matches the typographic mass of `font-size: 13px` line-height pills as styled in Dashboard_v2 `.pill` (line 203: `padding: 4px 10px` for 12px pills; we step up one for 13px pills).
- `.chat-summary-callout` uses `padding: 14px 18px` (research-recommended) — 14/18 is OFF-scale by 2px each, mirroring Dashboard_v2 `.msg-body` callouts (`padding: 14px 18px` at line 327). Phase 3 deliberately matches Dashboard_v2 here.
- `.chat-final-card .chat-plotly { margin-top: 18px }` — 18px is OFF-scale; 16px reads cramped under a 13px summary callout. Acceptable carry-forward of Dashboard_v2 spacing.

These three 2px-off values are visual-fidelity carry-forwards from Dashboard_v2 and are documented here so the executor does not "correct" them to scale-strict 16/24.

## Typography

Three sizes, two weights — minimal, anchored to `tokens.css` `--font-size-body: 15px` and the v2.0 type scale.

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Body | 15px (`--font-size-body`) | 400 | 1.5 | Default page body, summary callout text, idle textarea (inherits from `html, body`) |
| Chat-meta (pills + thoughts) | 13px | 500 (pills), 400 italic (thoughts) | 1.5 | `.chat-pill-tool-call`, `.chat-pill-tool-result-*`, `.chat-thought`, expanded SQL/JSON code blocks |
| Heading (panel-title) | 18px | 700 | 1.2 | `.panel-header .panel-title` ("Ask") — verbatim from Phase 02 D-UI2-12 |

**Mono font** (`"JetBrains Mono", ui-monospace, monospace`) is applied only to:
- `.chat-pill-tool-call` (the inline `run_sql("SELECT…")` summary line)
- Expanded SQL code blocks inside the tool_call expansion (sqlparse-formatted, `font-size: 13px`)
- `.pivot-table td` inside the final-card table (inherits Phase 4 D-27 verbatim)
- Expanded JSON args for non-SQL tool calls

Body text inside the summary callout, error cards, thoughts, and Stop button uses Inter Tight (default).

**No display size declared** — the Ask page does not use the `.page-title { 28px }` rule. The page heading lives inside `.panel-header` at 18px (`--font-size-body` step + 3px) per Phase 02 panel-header convention. This intentionally aligns Ask with Browse and Joint Validation (which use the same `.panel-header .panel-title` pattern post-Phase 02 D-UI2-12).

## Color

Project palette is locked in `tokens.css` (Phase 02). Dashboard_v2.html reservation rules below.

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `--bg: #f3f4f6` (page background) + `--panel: #ffffff` (chat panel) | Page surface; chat panel; transcript region; expanded code-block backgrounds |
| Secondary (30%) | `--ink: #171c24`, `--ink-2: #4a5361`, `--mute: #8b93a0`, `--line: #eaecef`, `--line-2: #e1e4e8` | Body ink; collapsed thought ink (`--mute`); thought left-border (`--line-2`); panel borders |
| Accent (10%) | `--accent: #3366ff` + `--accent-soft: #ebf1ff` + `--accent-ink: #1f4dcf` | Specific elements ONLY — see "Accent reserved for" below |
| Destructive | `--red: #ef3e4a` + `--red-soft: #fde8ea` + **`#cc2434` for AA-text variant** | REJECTED tool_result pill; hard error card border + ink; Stop button border + text (uses `#cc2434` to hit AA at 14px/600 — see §H) |
| Soft-warning | `--amber: #f59e0b` + `--amber-soft: #fef3d9` + **`#8a5a00` for AA-text variant** | Soft error card (timeout, step-budget-exhausted, stopped-by-user). Heading text uses `#8a5a00` ink (AA-passing on `--amber-soft`); border uses `--amber` (non-text). |
| AI-affordance | `--violet: #7a5af8` + `--violet-soft: #efebfe` | tool_call pills only |
| Success | `--green: #17a963` + `--green-soft: #e5f7ee` | Successful tool_result pills only |

**Accent reserved for** (explicit list — never "all interactive elements"):
1. The summary callout background at the top of `_final_card.html` (`background: var(--accent-soft)`, `color: var(--accent-ink)`). One per turn.
2. The "Run" submit button in the idle input form (`btn btn-primary` — Bootstrap default uses `--accent` shade, kept verbatim from Phase 6 D-13).
3. The active LLM dropdown item in the `.dropdown-menu` (Bootstrap `.active` modifier — verbatim from Phase 6).
4. The "Retry this question" button on the soft-error card (`btn btn-primary btn-sm` — same Bootstrap accent).
5. Cell hyperlinks if the pivot table inside the final card contains URL strings (inherits Phase 4 `.overview-table tbody td a { color: var(--accent) }`).

**Destructive (`--red` family) reserved for:**
1. The Stop button border + text ink + active hover bg. **Text and border use `#cc2434`** (AA-compliant red at 14px/600 on white — see §H rationale); hover background uses `--red-soft` (`#fde8ea`).
2. REJECTED tool_result pill background (`--red-soft`) + ink (`--red`). The pill text is 13px/500 on `--red-soft` background; this is decorative-grade affordance, not body copy — visual contract intentionally trades some contrast for hue identity (acceptable since the meaning is also encoded in the ▸ icon and "REJECTED:" copy prefix).
3. Hard-error card border + heading ink (LLM 5xx, retry-cap-exceeded, stream-dropped, agent-no-final-result). Heading uses `--red` at 14px/700 — passes AA Large (3.85:1 at large/bold per WCAG 2.1 §1.4.3 large-text definition: 14pt bold ≈ 18.66px / 19px CSS at weight 700, but 14px/700 sits at the threshold; body text uses `--ink` at 15.54:1 which carries the readable load). See §Accessibility.

**Soft-warning (`--amber` family) reserved for:**
1. Soft-error card border (uses `--amber` — non-text border, AA non-text 3:1 satisfied against `--bg` and `--panel` neighbors).
2. Soft-error card heading ink — uses **`#8a5a00`** (5.37:1 on `--amber-soft` — comfortable AA pass; hue still amber). **Do not use `--amber` itself for text on `--amber-soft`** — it computes to 1.95:1 contrast and fails AA at any size.

Per CONTEXT.md D-CHAT-04, soft errors are user-mitigatable boundaries, not application failures.

**Violet (`--violet`) reserved for:**
1. tool_call pills (only). Re-uses the existing `.ai-btn` violet aesthetic from `app.css` — same hue, but the pill is rounded fully (`--radius-pill: 999px`) instead of `7px`, and is mono-font instead of Inter Tight.

**Green (`--green`) reserved for:**
1. Successful tool_result pills (only).

| LLM dropdown empty | `No LLMs configured.` (verbatim from Phase 6) |

**Destructive actions** in this phase:
- **Stop the running turn** — single click, no confirmation dialog. Cooperative cancellation per D-CHAT-01; the running tool finishes before the loop exits, so the only "destruction" is canceling future planned tool calls. No confirmation needed; intent is unambiguous and the soft-error card "You stopped this turn." gives the user a clean recovery affordance via the `Retry this question` button.

There are no destructive actions on user data (DB is read-only; chat history is in-memory and ephemeral).

## Visual Contract — Per-Component Rules

Each rule below is **prescriptive**. The executor must implement these values exactly; the auditor compares to this section.

### A. Chat shell (`index.html`)

- Outer wrapper: `<div class="shell">` — Phase 02 full-width shell (`padding: 0`, no max-width).
- Single `.panel` wrapping the header and body. **No `.mb-3` on the panel** (chat is the only content on the page; bottom margin on the panel would create a dangling whitespace band against the sticky footer).
- Header: `.panel-header` with `<h1 class="panel-title">Ask</h1>` left + `.dropdown.ms-auto` right. **Verbatim from current Phase 6 `index.html` lines 43–84.** No layout changes; the LLM dropdown stays at panel-header far-right (D-CHAT-11 + Claude's-discretion default).
- Body: `.panel-body { padding: 26px 32px }` (Phase 02 verbatim) containing two children:
  1. `<div id="chat-transcript" hx-ext="sse" sse-swap="thought,tool_call,tool_result,final,error" hx-swap="beforeend" sse-close="final error">` — empty on initial render. Min-height: **none** (let it grow from 0 as events arrive — no skeleton placeholder).
  2. `<div id="input-zone">{% include "ask/_input_zone.html" %}</div>` — always present; toggles between idle and active states.
- Plotly bundle: `<script src="{{ url_for('static', path='vendor/plotly/plotly.min.js') }}" defer></script>` placed at the END of `{% block content %}` (NOT in `{% block extra_head %}` — the bundle is large and `defer` on a body-tail tag delays parsing more cleanly than a deferred head tag for the Ask page only).

### B. User message + thinking placeholder (`_user_message.html`)

- Single block, top of transcript. Visually quiet — no bubble background, no avatar (intranet shared-creds, no per-user identity).
- Render: `<div class="chat-user-message">` containing `<p class="chat-user-question">{{ question | e }}</p>` followed by `<p class="chat-thinking">Thinking…</p>`.
- `.chat-user-question`: `font-size: 15px`, `font-weight: 600`, `color: var(--ink)`, `margin: 0 0 8px`. Emulates "you asked:" but without the prefix label (the question alone, in semibold, reads as the user's voice).
- `.chat-thinking`: `font-size: 13px`, `font-style: italic`, `color: var(--mute)`, `margin: 0 0 16px`. Three trailing dots are the literal `…` (U+2026), no animation in Phase 3 (motion is a Phase 4+ enhancement). Removed when first SSE `thought` or `tool_call` event arrives.
- **No motion required** for Phase 3. The presence of the SSE event stream + browser EventSource API gives the perception of motion via incremental DOM appends.

**Why `#8a5a00` ink for the soft heading:** `--amber: #f59e0b` against `--amber-soft: #fef3d9` is only **1.95:1** contrast — fails WCAG AA at any size (motivation only; this pair is never used). Darkening the heading ink to `#8a5a00` (HSL hue preserved, lightness lowered) computes to **5.37:1** on `--amber-soft` and passes WCAG 2.1 AA for normal text (≥4.5:1 required). The body text uses `--ink` (`#171c24`) on `--amber-soft` which computes to **14.79:1** — comfortable AAA. Border stays at `--amber` (`#f59e0b`) since border contrast is non-text and AA only requires 3:1 against the neighboring `--bg`/`--panel` surfaces (passes).

> Math reference (per WCAG 2.1 §1.4.3 relative-luminance formula): convert sRGB 0..255 to 0..1, apply `c <= 0.03928 ? c/12.92 : ((c + 0.055) / 1.055) ** 2.4` per channel, then `L = 0.2126*R + 0.7152*G + 0.0722*B`; contrast = `(L_lighter + 0.05) / (L_darker + 0.05)`. The previous draft cited `#b27500` at "5.13:1" — that pair actually computes to 3.50:1 and fails AA; replaced with `#8a5a00`.

## Interaction & Motion

| Interaction | Trigger | Visual effect | Duration | Easing |
|-------------|---------|---------------|----------|--------|
| Submit question | Click "Run" or Cmd/Ctrl-Enter in textarea | `_user_message.html` fragment appended to `#chat-transcript`; `_input_zone.html` swapped to Stop state via OOB | instant | n/a |
| SSE event arrives (any) | EventSource onmessage | New fragment appended to `#chat-transcript` (no animation) | instant | n/a |
| Expand thought / pill | Click `<details>` summary | Native browser `<details>` toggle | browser-default (~150ms slide) | browser-default |
| Stop button hover | Pointer enter | Background fades from `transparent` to `--red-soft` | 120ms | ease |
| Stop button press | Click | Background = `#fbd2d6` while held | instant | n/a |
| Stop click | Click | `_input_zone` flips to disabled-ish state visually (button greys); SSE eventually emits `error` (stopped-by-user); transcript receives soft-error card; `_input_zone` flips back to idle via OOB | governed by SSE event arrival | n/a |
| LLM dropdown selection | Click dropdown item | 204 + HX-Refresh: true → full page reload | instant page reload | browser-default |
| Final card arrival | SSE `final` event | Final card fragment appended; SSE listener closes (`sse-close="final error"`); `_input_zone` swaps back to idle via OOB inside the final-card fragment | instant | n/a |
| Error card arrival | SSE `error` event | Same as final card but the rendered fragment is `_error_card.html` | instant | n/a |

**Phase 3 motion is intentionally minimal.** The "agentic feel" comes from the live arrival of SSE events, not from CSS animation. Adding a fade-in or scale-in to chat events would compete with the natural SSE pacing and add a perceived latency floor. Future phases may explore a subtle 80ms fade-in on `.chat-pill-*` if user feedback demands it.
