DIAGNOSIS — Why text_delta SSE never fires in practice:

  app/core/agent/chat_agent.py:194-208 constructs the agent with
  output_type=ToolOutput(PresentResult, name="present_result"). With ToolOutput,
  PydanticAI requires the model to call the present_result tool to terminate.
  In practice, OpenAI (and most Ollama models that support tool calling)
  emit ALL output through the tool-call channel as ToolCallPartDelta — they
  do NOT emit standalone TextPart / TextPartDelta events when a tool-call is
  forced/expected. The free-text "preamble" assumption from 260504-114 is
  rare-to-never in production. The text_delta plumbing in chat_loop._event_to_payload
  is correct but the events never arrive. This is a PROVIDER + STRUCTURED-OUTPUT
  interaction, not a wire-format or sse-swap bug:

    - sse-swap="thought,tool_call,tool_result,text_delta,final,error" is correct
    - hx-swap="beforeend" on .chat-events is correct (would accumulate text_delta
      spans if any arrived)
    - sse-starlette ServerSentEvent.encode() correctly splits multi-line data
      into multiple "data: " lines per the SSE spec
      (verified at .venv/lib/python3.13/site-packages/sse_starlette/event.py:49-51)
    - chat_loop._event_to_payload correctly handles PartStartEvent(TextPart)
      with non-empty content AND PartDeltaEvent(TextPartDelta)
    - app_v2/templates/ask/_text_delta.html correctly returns single-line
      <span class="chat-text-delta">{{ delta | e }}</span>

  CONCLUSION: text_delta works correctly when fed events; the agent simply
  never feeds it events. No diagnostic logging task is needed — the structured-

WORD-REVEAL ANIMATION (Concern 3 — perceived streaming):

  Server-side: _final_card.html splits the summary string on whitespace and
  wraps each token in a <span class="sw">{{ token | e }}</span>, with a literal
  space character between spans (so word boundaries survive). The wrapping is
  done in Jinja using {{ summary.split() }} via a {% for %} loop — no Python
  helper needed. Each .sw span starts at opacity 0 (CSS default) and animates
  to opacity 1 when the .is-visible class is added.

  Client-side: a new IIFE in index.html listens for htmx:sseMessage with
  evt.detail.type === 'final', then queries the most-recently-mounted
  .chat-summary-callout (the LAST one in the document) and runs a setInterval
  at ~30ms cadence that adds .is-visible to one .sw span at a time, in
  document order, until all are visible. Cadence × word count caps total
  reveal at ~1-2 seconds for a typical 1-2 sentence summary.

  Why client-side and not server-side splitting via text_delta SSE:
    - We could split summary into chunks and emit them as text_delta events
      BEFORE the final event, but that would require re-architecting the
      structured-output contract (the final card carries summary + sql +
      chart_spec atomically; splitting summary out of the final payload
      would break _hydrate_final_card's render contract).
    - Client-side reveal is fully decoupled from the SSE protocol. It works
      identically for OpenAI and Ollama. It runs after the final-card markup
      is in the DOM, so there's no race with htmx swap timing.
    - Reduced-motion: a single CSS rule
        @media (prefers-reduced-motion: reduce) { .sw { opacity: 1; } }
      forces all words visible immediately, no animation. The JS handler
      still runs and adds .is-visible classes (no harm done — they're
      idempotent and CSS already shows them at opacity 1).

```html
  <script>
    // Quick task 260504-220 — perceived streaming for the assistant summary.
    //
    // The agent uses ToolOutput(PresentResult), so OpenAI/Ollama deliver the
    // summary atomically via the 'final' SSE event (no token-level streaming
    // upstream). Rather than fight the structured-output contract, we
    // reveal the summary text word-by-word client-side after the final card
    // mounts: every <span class="sw"> in .chat-summary-callout starts at
    // opacity 0; we add .is-visible to each on a 30ms timer in document
    // order. The CSS transition fades each word in over 80ms. A typical
    // 1-2 sentence summary completes in 600-1200ms — fast enough to feel
    // "alive", slow enough that the eye reads it as progressive disclosure.
    //
    // Reduced-motion: CSS already forces .sw { opacity: 1 } in the
    // prefers-reduced-motion media query (see app.css). The IIFE still runs
    // and adds .is-visible classes (no-op since opacity is already 1), so
    // the timing fires without producing any visible motion — accessible
    // by construction.
    //
    // Idempotency: guarded by window.__pbm2RevealSummaryBound so re-renders
    // can't double-bind. The handler scopes its action to the LAST
    // .chat-summary-callout in the document — that's the one that just
    // mounted, since per-turn final cards append at the end of the
    // transcript.
    (function () {
      if (window.__pbm2RevealSummaryBound) return;
      window.__pbm2RevealSummaryBound = true;
      var REVEAL_INTERVAL_MS = 30;
      document.body.addEventListener("htmx:sseMessage", function (evt) {
        // evt.detail.type holds the SSE event name (htmx-ext-sse 2.x).
        // Filter so this only runs on 'final' events; thought/tool_call/
        // tool_result/text_delta/error each fire htmx:sseMessage too and
        // we want the reveal to run exactly once per turn.
        var detail = evt.detail || {};
        if (detail.type !== "final") return;
        // Wait one tick so htmx finishes inserting the final-card markup
        // before we walk it. The final card mounts via beforeend swap on
        // .chat-events; htmx fires sseMessage AFTER the swap completes,
        // but defensively we use rAF to guarantee the DOM is settled.
        requestAnimationFrame(function () {
          var callouts = document.querySelectorAll(".chat-summary-callout");
          if (callouts.length === 0) return;
          var callout = callouts[callouts.length - 1];   // most recently mounted
          var spans = callout.querySelectorAll(".sw");
          if (spans.length === 0) return;
          var i = 0;
          var timer = setInterval(function () {
            if (i >= spans.length) {
              clearInterval(timer);
              return;
            }
            spans[i].classList.add("is-visible");
            i += 1;
          }, REVEAL_INTERVAL_MS);
        });
      });
    })();
  </script>
```
