# Phase 3 — UI Design Contract: Content Pages + AI Summary

> Visual and interaction contract for the platform detail page, markdown CRUD, and in-place AI Summary.
> Generated by gsd-ui-researcher. Verified by gsd-ui-checker.
>
> **Design system:** Bootstrap 5.3.8 (vendored) + a thin **Dashboard token layer** introduced in this phase
> (`app_v2/static/css/tokens.css` + `app_v2/static/css/app.css`). All visual treatment anchors to
> `Dashboard_v2.html` (the canonical visual reference). Bootstrap utilities remain the primary layout
> mechanism; tokens supply color, radius, and the `.ai-btn` / `.panel` patterns Bootstrap cannot express.

---

## Design System

| Property | Value |
|----------|-------|
| Tool | none (Bootstrap 5.3.8 vendored CSS + JS + Dashboard token layer) |
| Preset | not applicable |
| Component library | Bootstrap 5 — nav-pills, btn, alert, spinner-border, form-control, details/summary |
| Icon library | Bootstrap Icons 1.13.1 (vendored) |
| Body font | `"Inter Tight", system-ui, -apple-system, "Segoe UI", Roboto, sans-serif` (15px base, letter-spacing -0.005em) |
| Mono font | `"JetBrains Mono", ui-monospace, monospace` (textarea, code blocks, summary metadata) |
| Token layer | `app_v2/static/css/tokens.css` (CSS custom properties) + `app_v2/static/css/app.css` (`.panel`, `.ai-btn`, `.markdown-content`) |

**Source:** Design Anchor (Dashboard_v2.html) + INFRA-04 (REQUIREMENTS.md) — Bootstrap 5.3.8, HTMX 2.0.10,
Bootstrap Icons 1.13.1 all vendored; no CDN. Phase 03 introduces the token layer; the Dashboard fonts are
loaded **only via the existing vendored stack as a fallback** — Inter Tight is loaded from Google Fonts
when network is available, otherwise system-ui takes over (intranet-safe degradation).

**Font loading strategy (intranet-safe):**

```html
<link rel="preconnect" href="https://fonts.googleapis.com">
<link href="https://fonts.googleapis.com/css2?family=Inter+Tight:wght@400;500;600;700;800&family=JetBrains+Mono:wght@400;500;600&display=swap" rel="stylesheet" media="print" onload="this.media='all'">
```

`media="print"` then JS swap to `'all'` makes the font request non-render-blocking; if the intranet has no
outbound DNS, the page still renders with system-ui (per INFRA-04 vendor-only requirement). The
`font-family` declaration in `tokens.css` lists Inter Tight first then a system stack so degradation is
graceful.

## Spacing Scale

Bootstrap 5 base-4 scale (`$spacer = 1rem = 16px`) is the source of truth. Phase 03 adds two **token-level
inner paddings** (Dashboard `.panel` has 18-26px padding which does not map to a Bootstrap utility).

| Token / Class | Value | Usage in Phase 3 |
|---------------|-------|------------------|
| `gap-1` | 4px | Sparkle icon → label gap inside `.ai-btn` |
| `gap-2` | 8px | Toolbar button group gap (Edit / Delete / AI Summary) |
| `mb-2` | 8px | Below page title row |
| `mb-3` | 16px | Between `.panel` cards |
| `p-3` | 16px | Edit textarea inner padding |
| `--panel-pad-y: 18px` (custom) | 18px | `.panel` vertical padding (Dashboard `.ph` row) |
| `--panel-pad-x: 26px` (custom) | 26px | `.panel` horizontal padding (Dashboard `.ph` row) |
| `py-4` | 24px | Empty state vertical breathing room |
| `py-5` | 48px | Full-panel empty state ("No content yet") |
| `mt-3` | 16px | Main content top margin (inherited from `base.html`) |
| Article max-width | 800px | Rendered markdown content centered inside `.panel` (D-03) |
| Page shell max-width | 1280px | `.shell` outer container — matches Dashboard (D-02 implicit) |

**Exceptions:**
- `.panel` inner padding `18px 26px` (vertical/horizontal asymmetric) is not a Bootstrap utility — applied via the `.panel` class in `app.css`. Source: Dashboard `.ph` rule.
- Article content area uses `max-width: 800px; margin: 0 auto; line-height: 1.6;` per D-03 — these are content-readability values, not layout spacing.

## Color

Phase 03 introduces a **token-level palette** alongside Bootstrap. The 60/30/10 split is made explicit
against the Dashboard tokens; Bootstrap semantic classes (`btn-primary`, `text-danger`) remain the
day-to-day vocabulary. Custom `.ai-btn` / `.panel` use tokens directly.

| Role | Token / Bootstrap | Computed | Usage |
|------|------------------|----------|-------|
| Dominant (60%) — page surface | `var(--bg)` | `#f3f4f6` | Body background — set on `body` in `tokens.css` |
| Secondary (30%) — card/panel surfaces | `var(--panel)` (white) | `#ffffff` | `.panel` background, navbar (inherited), edit textarea wrapper |
| Accent (10%) — primary actions + Bootstrap mappings | `var(--accent)` ↔ `btn-primary` | `#3366ff` | Save button, "Add Content" empty-state CTA, Brand badge (inherited Phase 2) |
| AI accent — violet | `var(--violet)` / `var(--violet-soft)` | `#7a5af8` / `#efebfe` | `.ai-btn` text + gradient bg, Regenerate button, AI metadata footer |
| Destructive | `var(--red)` ↔ `btn-outline-danger` | `#ef3e4a` | Delete button, error retry icon |
| Warning (recoverable) | `var(--amber)` ↔ `alert-warning` | `#f59e0b` | LLM failure alert (D-16) |
| Success | `var(--green)` ↔ `bg-success` | `#17a963` | Year badge (inherited), save-success toast (Bootstrap default mapping) |
| Muted / helper | `var(--mute)` ↔ `text-muted` | `#8b93a0` | Empty state copy, summary metadata, breadcrumb-like context |
| Border / divider | `var(--line)` | `#eaecef` | Hairlines inside `.panel`, between summary metadata and body |

**Accent (#3366ff) reserved exclusively for:**
1. **Save** button on the edit view (`btn btn-primary`)
2. **Add Content** button on the empty state (`btn btn-primary`)
3. **Active** nav-link (inherited from `base.html` Phase 1)
4. **Brand** badge inherited from Phase 02 (`badge rounded-pill bg-primary`)
5. **Active** nav-pill (Write or Preview tab — Bootstrap `.nav-pills .nav-link.active` background defaults to `--bs-primary`; we keep it on `var(--accent)`)
6. Inline links inside rendered markdown (`<a>` default color → `var(--accent)`)

**Violet (#7a5af8) reserved exclusively for AI affordances:**
1. `.ai-btn` (Overview row + detail page header) — gradient pill
2. `.ai-btn.regen` Regenerate button — same gradient, sparkle icon flipped
3. AI summary "Summarizing… (using {backend})" loading text color
4. AI metadata footer (`text-violet-mute` — `var(--violet)` at 50% opacity)

**Why violet ≠ accent:** Dashboard reserves violet for the AI affordance specifically so LLM-touched
surfaces are visually distinct from regular CTAs. A user must always be able to tell at-a-glance "this
button calls a language model." Never use violet for non-AI buttons in this phase.

**Destructive (red) reserved exclusively for:**
1. Delete button on detail page header (`btn btn-outline-danger` with `bi-trash3`)
2. (No others in Phase 03 — the Remove button on Overview rows is inherited from Phase 02 and stays as-is)

**Warning (amber) reserved exclusively for:**
1. LLM failure alert in summary slot (`.alert .alert-warning`)
2. Retry button inside that alert (`btn btn-sm btn-outline-warning`)

**Why warning, not danger, for LLM failure:** Per D-16 — LLM failure is recoverable (network blip, model
overloaded). Red would suggest data loss or destructive action; amber correctly signals "transient,
retry."

All major content surfaces sit inside a `.panel` (rendered markdown, edit view, empty state).

```css
.panel {
```

**Why bigger inner padding here than Dashboard's `.ph` (18/26)** — content reading benefits from more

800px-wide article visible breathing room on a 1280px shell.

### 4. Rendered Markdown View (Detail Page, `has_content == true`)

The persistent ID `#content-area` is the **single HTMX swap target** that toggles between rendered view,
edit view, and empty state. Server returns the appropriate full fragment for each transition.

```html
<div class="panel" id="content-area">
  <div class="panel-body">
    <article class="markdown-content">
      {{ rendered_html | safe }}  {# Rendered by MarkdownIt("js-default") — XSS-safe per D-08 #}
    </article>
  </div>
</div>

{# AI Summary slot lives BELOW the content panel. Empty until the button is clicked. #}
<div id="summary-{{ platform_id }}" class="mt-3"></div>
```

**Markdown content max-width and line-height:** `max-width: 800px; line-height: 1.6;` (D-03).

**`{{ rendered_html | safe }}` is mandatory** — the value comes from `MarkdownIt("js-default")` which has
already escaped raw HTML; Jinja's autoescape would double-escape and break rendering. The `js-default`
preset (NOT default-with-html) is the XSS defense (D-08, T-03-02).

**Source:** D-02, D-03, D-08 (CONTEXT.md), CONTENT-03 (REQUIREMENTS.md).

**Custom `app.css` for nav-pills to pin Dashboard accent:**

```css
.nav-pills .nav-link {

  padding: 6px 14px;
  font-size: 14px;
```

**Why nav-pills (not nav-tabs):** D-06 explicitly chose pills; Dashboard uses pill-rounded inactive

**Cancel is client-side** (D-10) — `data-cancel-html` attribute holds the original rendered-view HTML;
`hx-on:click` swaps it back. No server round-trip. The server populates `data-cancel-html` with either
the rendered article HTML (existing content) or the empty-state HTML (new content), both already escaped
via Jinja `| e` filter so attribute injection is impossible.

**Save returns the rendered-view fragment** (D-09) — `outerHTML` swap on `#content-area` replaces the
entire edit panel with the rendered markdown panel.

**Preview hx-trigger:** `click, keyup changed delay:500ms from:#md-textarea` — clicking the Preview pill
fetches; typing in the textarea while the Preview tab is active triggers the same fetch debounced 500ms
(D-07).

**Char counter:** Updates client-side via a tiny inline JS `oninput` — counts textarea length, color
turns `var(--red)` at >65000 chars. Defensive UX for the 64KB cap (D-31).

**Source:** D-06, D-07, D-09, D-10 (CONTEXT.md), CONTENT-04, CONTENT-05, CONTENT-06, CONTENT-07
(REQUIREMENTS.md).

### 8. AI Summary Slot — Loading / Success / Error States

The `<div id="summary-{pid}">` is initially empty. HTMX swaps the appropriate fragment in on each click.

#### 8a. Loading State (D-14)

```html
<div class="panel mt-2" style="padding: 16px 20px;">
  <div class="d-flex align-items-center gap-2">
    <div class="spinner-border spinner-border-sm" role="status" style="color: var(--violet);">
      <span class="visually-hidden">Loading…</span>
    </div>
    <span style="color: var(--violet); font-weight: 500;">Summarizing… (using {{ backend_name }})</span>
  </div>
</div>
```

Server-side rendering rule: `backend_name` resolves to `"Ollama"` or `"OpenAI"` from
`app.state.settings.llm.type` capitalized.

**Why the request returns the loading fragment first?** It does NOT — HTMX 2.0.10's `htmx-request` class
is applied automatically to the trigger element (button) during the in-flight request, but the **target
slot remains empty** until the response arrives. The "loading" fragment shown above is **not used**;
instead we render the loading affordance **inside the button** via the `.ai-btn.htmx-request` CSS rule
(opacity 0.6 + spinner injection).

**Correction — actual loading pattern (D-14, definitive):** HTMX's standard `htmx-indicator` pattern
applies. The summary slot pre-seeds an indicator that is hidden by default and revealed via the
`htmx-request` class on the parent. The single source of truth for the loading affordance lives **inside
the slot itself**, swapped in immediately by an HTMX `hx-swap-oob="afterbegin"` placeholder OR via the
button's `hx-indicator` attribute pointing at `#summary-{pid}`. We use the `hx-indicator` approach:

```html
{# Button declares which element shows the spinner #}
<button class="ai-btn"
        hx-post="/platforms/{{ pid }}/summary"
        hx-target="#summary-{{ pid }}"
        hx-swap="innerHTML"
        hx-indicator="#summary-{{ pid }}-spinner"
        hx-disabled-elt="this">
  <i class="bi bi-stars"></i> AI Summary
</button>

{# Slot pre-renders an invisible spinner that becomes visible during the request #}
<div id="summary-{{ pid }}" class="mt-3">
  <div id="summary-{{ pid }}-spinner" class="htmx-indicator panel mt-2 d-flex align-items-center gap-2" style="padding: 16px 20px;">
    <div class="spinner-border spinner-border-sm" role="status" style="color: var(--violet);">
      <span class="visually-hidden">Loading…</span>
    </div>
    <span style="color: var(--violet); font-weight: 500;">Summarizing… (using {{ backend_name }})</span>
  </div>
</div>
```

`htmx-indicator` class is hidden by default; HTMX automatically toggles `htmx-request` during the
in-flight request, which the indicator CSS reacts to. When the response arrives, `innerHTML` swap
**replaces** the entire slot contents (including the spinner) with the success or error fragment. This
pattern is HTMX 2.0.10 idiomatic and matches what the Phase 02 spec already established for fragment
swaps.

#### 8b. Success State (D-15)

```html
<div class="panel mt-2">
  <div class="panel-body" style="padding: 18px 22px;">
    <article class="markdown-content" style="max-width: 720px; line-height: 1.55;">
      {{ summary_html | safe }}  {# 2-3 markdown bullets rendered via the same MarkdownIt pipeline #}
    </article>
    <div class="d-flex justify-content-between align-items-center mt-2 pt-2" style="border-top: 1px solid var(--line);">
      <small class="text-muted">
        <span class="mono">{{ llm_name }} · {{ llm_model }}</span>
        · cached {{ cached_age_s }}s ago
      </small>
      <button class="ai-btn regen"
              hx-post="/platforms/{{ pid }}/summary"
              hx-target="#summary-{{ pid }}"
              hx-swap="innerHTML"
              hx-indicator="#summary-{{ pid }}-spinner"
              hx-headers='{"X-Regenerate": "true"}'
              hx-disabled-elt="this"
              title="Regenerate ignoring cache">
        <i class="bi bi-arrow-clockwise"></i> Regenerate
      </button>
    </div>
  </div>
</div>
```

**Metadata footer** (Claude's Discretion accepted, per CONTEXT.md "recommend yes for trust + cost
transparency"): shows the LLM backend name + model + cache age. `cached_age_s` is `0` for a fresh call
and the actual age (in seconds) for cache hits — gives users a "this is from cache" signal.

**Regenerate icon:** `bi-arrow-clockwise` (NOT `bi-stars`) so the user can distinguish the action from a
fresh AI Summary at a glance. Title attribute: `"Regenerate ignoring cache"` (per Claude's Discretion in
CONTEXT.md).

#### 8c. Error State (D-16)

```html
<div class="alert alert-warning d-flex justify-content-between align-items-center mt-2" role="alert">
  <div>
    <i class="bi bi-exclamation-triangle me-2"></i>
    Summary unavailable: {{ reason }}. Try again or switch LLM backend in Settings.
  </div>
  <button class="btn btn-sm btn-outline-warning"
          hx-post="/platforms/{{ pid }}/summary"
          hx-target="#summary-{{ pid }}"
          hx-swap="innerHTML"
          hx-indicator="#summary-{{ pid }}-spinner"
          hx-disabled-elt="this">
    <i class="bi bi-arrow-clockwise"></i> Retry
  </button>
</div>
```

**`{{ reason }}` is server-classified, not a raw exception message** — the route maps exceptions to
user-readable strings:

| Exception class | `reason` string |
|----------------|----------------|
| `httpx.ConnectError`, `httpx.ConnectTimeout` | `Cannot reach the LLM backend ({backend_name})` |
| `httpx.ReadTimeout`, `asyncio.TimeoutError` | `LLM took too long to respond` |
| `openai.AuthenticationError` | `LLM authentication failed — check API key in Settings` |
| `openai.RateLimitError` | `LLM is rate-limited — try again in a moment` |
| `openai.APIStatusError` (5xx) | `LLM backend returned an error (HTTP {status})` |
| `FileNotFoundError` (content disappeared mid-call) | `Content page no longer exists` |
| Any other | `Unexpected error — see server logs` |

This is the **complete error vocabulary** for Phase 03. No raw exception traces leak to the UI (T-03-06
adjacent — defense in depth).

**Why amber, not red:** D-16 — recoverable failure. The Retry button is the affordance that completes the
visual "transient problem" message.

**Source:** D-14, D-15, D-16, D-17, D-18, D-21 (CONTEXT.md); SUMMARY-03, SUMMARY-04, SUMMARY-05, SUMMARY-06, SUMMARY-07 (REQUIREMENTS.md).

## HTMX Behavior Summary

| Element | Trigger | Method + Endpoint | Target | Swap |
|---------|---------|-------------------|--------|------|
| AI Summary button (Overview or detail) | click | `POST /platforms/{pid}/summary` | `#summary-{pid}` | `innerHTML` |
| Regenerate button | click | `POST /platforms/{pid}/summary` (header `X-Regenerate: true`) | `#summary-{pid}` | `innerHTML` |
| Retry button (error alert) | click | `POST /platforms/{pid}/summary` | `#summary-{pid}` | `innerHTML` |
| Edit button | click | `POST /platforms/{pid}/edit` | `#content-area` | `outerHTML` |
| Add Content button (empty state) | click | `POST /platforms/{pid}/edit` | `#content-area` | `outerHTML` |
| Save button | submit | `POST /platforms/{pid}` | `#content-area` | `outerHTML` |
| Cancel button | click | (client-side) — restores `data-cancel-html` | `#content-area` | (none — DOM swap inline) |
| Preview tab | click + keyup-debounced 500ms | `POST /platforms/{pid}/preview` | `#preview-pane` | `innerHTML` |

| Char counter | input event (vanilla JS) | (none) | (none) | (none) |
| Errors (4xx/5xx) | automatic (htmx-error-handler.js, Phase 1) | — | `#htmx-error-container` | `innerHTML` |

**`hx-disabled-elt="this"` is mandatory** on every AI Summary trigger to prevent double-submit during
LLM in-flight (SUMMARY-02). All endpoints that mutate disk state (POST `/platforms/{pid}`, DELETE
`/platforms/{pid}/content`) also use `hx-disabled-elt="this"` (defensive — though atomic write protects
against partial state, the user shouldn't see a flicker of a re-submitted form).

**No URL pushState in Phase 03** — `hx-push-url` is NOT applied on edit/save/preview (the URL stays at
`/platforms/{pid}`; the visual state inside is what changes). Browser back from the edit view goes back to
the prior page (Overview), not to the rendered view (D-05). This is the correct contract: edit/preview/save
is a **modal-like inline state**, not a route.

## State Diagram — Detail Page `#content-area`

The single `#content-area` element transitions through 3 stable states + 1 transient state:

```
[Empty state]          ←──────────[Delete success]──────────────┐
       │                                                          │
   [Add Content] click                                            │
       ↓                                                          │
[Edit view (new)] ────[Save]────→ [Rendered view] ──[Edit click]──┘
       │                              │                           │
       └─[Cancel client-side]────→ [Empty state]                  │
                                                                  │
                                  [Save invalid 422]              │
                                       ↓                          │
                                  Error to #htmx-error-container, │
                                  edit view stays                 │
```

**Each transition is a single HTMX swap on `#content-area` with `outerHTML`** — this means the server
returns a complete new `.panel` fragment, never a partial. The `data-cancel-html` attribute on the edit
panel persists across the **edit → preview → edit** internal navigation (Bootstrap pill toggle, no swap),
and is consumed once by Cancel.

## State Diagram — Summary Slot `#summary-{pid}`

```
[Empty]
   │
[AI Summary click] ──→ [Loading spinner] ──┬──→ [Success card + Regenerate]
```

## Accessibility Checklist

| Requirement | Implementation |
|-------------|---------------|
| Icon-only buttons have aria-label | All header buttons have `aria-label` (Edit/Delete/AI Summary on detail page); Phase 02 inherited Remove (×) keeps its label |
| Active nav-pill marked | `data-bs-toggle="pill"` + Bootstrap auto-applies `aria-selected` on the `.nav-link` |
| Tab panes labelled | `role="tabpanel" aria-labelledby="write-tab"` etc. on each `.tab-pane` |
| Form inputs labelled | Textarea has `id="md-textarea"` and a `<span class="visually-hidden">Markdown content</span>` paired via `aria-labelledby` (added in template); placeholder is supplemental, not the label |
| Alert roles | All alerts carry `role="alert"`; spinners carry `role="status"` |
| Disabled button communicates state | `disabled` attribute + descriptive `title` on AI Summary disabled state |
| Loading state announced | `<span class="visually-hidden">Loading…</span>` inside spinner; the visible "Summarizing…" text is announced by screen readers as a live region (we add `aria-live="polite"` to `#summary-{pid}`) |
| Color is not the sole differentiator | Error alert has `bi-exclamation-triangle` icon + amber border + "Summary unavailable" prefix text; success has rendered text content; no color-only signaling |
| Focus management on swap | After Save success, focus returns to the Edit button (server adds `hx-on::after-swap="document.querySelector('[hx-post*=edit]')?.focus()"` on the rendered view) |
| `hx-confirm` accessible | Native `window.confirm()` is accessible to screen readers |
| Markdown preview iframe | NOT used — `<article>` direct rendering keeps focus management simple and screen-reader-natural |
| Keyboard nav for tabs | Bootstrap nav-pills with `data-bs-toggle="pill"` provides arrow-key + Enter/Space behavior natively |
| `font-size: 15px` base | Above the 14px WCAG comfortable-reading floor; rendered article uses 15px body / 1.6 line-height — well-spaced |

**`aria-live="polite"` on summary slot:** The `#summary-{pid}` wrapper carries `aria-live="polite"
aria-atomic="true"` so screen readers announce the loading text and the eventual summary content. Polite
(not assertive) — this is informational, not an error.
