## Shell Layout Contract

### Nav Bar (D-UI2-01, D-UI2-02)

**Selector:** `.navbar.navbar-light.bg-light.border-bottom` (existing) — padding override in app.css.

```css
/* New rule in app.css §1 Nav Bar */
.navbar { padding-top: 16px; padding-bottom: 16px; }
```

Nav structure (base.html restructure):
```
[navbar-brand: bi-database icon + "PBM2"]  [nav-tabs: Joint Validation | Browse | Ask]
← left-aligned, ms-3 gap between brand and tabs                    right side: empty
```

Current `<ul class="nav nav-tabs ... ms-3">` already has `ms-3` (left margin from brand) — the tabs are effectively left-aligned already in the current markup. Confirm `ms-auto` is NOT on the `<ul>`. Current base.html inspection confirms no `ms-auto` on the nav-tabs ul. **No structural change needed for tab alignment** — the current markup already places tabs left. Only the padding increase (D-UI2-02) is the shell nav change.

### Full-Width Content (D-UI2-03)

**Selector:** `.shell` in app.css.

Old rule: `.shell { max-width: 1280px; margin: 0 auto; padding: 18px 24px 56px; }`
New rule: `.shell { padding: 0; }` — remove max-width, margin, and outer padding entirely.

The `<main class="container-fluid mt-3">` in base.html already stretches full-width. The `.shell` div inside each page's `{% block content %}` inherits full width when its own constraints are removed. The `mt-3` on `<main>` (12px top margin from Bootstrap utility) is preserved.

### Sticky In-Flow Footer (D-UI2-05)

**Pattern:** Flex column on body with `min-height: 100vh`; footer pushed via `margin-top: auto`.

```css
/* New rules in app.css §0 Body Flex Shell */
body {
  display: flex;
  flex-direction: column;
  min-height: 100vh;
}
main.container-fluid {
  flex: 1 0 auto;
}
.site-footer {
  flex-shrink: 0;
  background: var(--panel);
  border-top: 1px solid var(--line);
  min-height: 48px;
  padding: 0 32px;
  display: flex;
  align-items: center;
  gap: 16px;
}
```

Footer slot structure in `base.html` (new block appended before `</body>`):
```html
<footer class="site-footer" id="site-footer">
  {% block footer %}{% endblock footer %}
</footer>
```

- Full-page renders (GET /browse, GET /overview, GET /ask, GET /joint_validation/<id>) extend this block.
- HTMX partial fragment renders (POST /browse/grid, POST /overview/grid) do NOT touch the footer — the OOB swap pattern updates footer content via a separate OOB block (see Browse Count section below).
- Detail pages and Ask page: `{% block footer %}{% endblock %}` — footer is rendered empty (shows as a 48px white stripe at page bottom, giving the rounded shell shape D-UI2-05 requires).

### Shell Rounded-Card Shape

The sticky footer + top nav form the "card" silhouette:
- Top: `<nav>` with `border-bottom: 1px solid var(--line)` and `bg-light` — forms the cap.
- Middle: `<main>` — content stretches.
- Bottom: `.site-footer` with `border-top: 1px solid var(--line)` and `background: white` — forms the base.
- The `.panel` inside uses `border-radius: var(--radius-panel)` (22px) — inner card shape unchanged.

### Filter Row Layout (D-UI2-08, D-UI2-09, D-UI2-10)

**CSS change to `.overview-filter-bar`** in app.css:

Old rule:
```css
.overview-filter-bar {
  padding: 0 26px;
}
```

New rule:
```css
.overview-filter-bar {
  padding: 12px 26px 0;
  border-bottom: 1px solid var(--line);
  display: flex;
  align-items: center;
  gap: 8px;
  flex-wrap: wrap;
}
```

- `display: flex` makes the six `<div class="dropdown">` elements (each currently block) line up horizontally.
- `gap: 8px` — matches Browse's `gap-2` (8px) between pickers.
- `flex-wrap: wrap` — allows graceful overflow on narrow viewports rather than horizontal scroll.
- `padding: 12px 26px 0` — matches Browse's `.browse-filter-bar { padding: 12px 26px 0 }`.
- `border-bottom: 1px solid var(--line)` — matches Browse's filter bar bottom separator.

**The form tag** (`<form id="overview-filter-form">`) must be placed BEFORE the filter bar include (Pitfall 4), as a `visually-hidden` / `aria-hidden` ghost form or embedded as the flex container. **Resolution:** The form becomes the flex container itself (wraps the six pickers + "Clear all"). This avoids a separate ghost form. The `<form>` gets the same `display: flex; gap: 8px; align-items: center; flex-wrap: wrap; width: 100%;` so it stretches:

```html
<div class="overview-filter-bar">
  <form id="overview-filter-form"
        class="d-flex align-items-center gap-2 flex-wrap w-100"
        hx-post="/overview/grid"
        hx-target="#overview-grid"
        hx-swap="outerHTML"
        hx-trigger="change delay:250ms from:find input[type=checkbox]"
        hx-include="this">
    <input type="hidden" name="sort" value="{{ vm.sort_col | e }}">
    <input type="hidden" name="order" value="{{ vm.sort_order | e }}">
    <input type="hidden" name="page" value="1">  {# reset to page 1 on filter change #}
    {{ picker_popover(...) }}  {# × 6 pickers #}
    <a href="/overview" class="ms-auto btn btn-link btn-sm ..."
       hx-get="/overview" hx-target="body" hx-push-url="true">Clear all</a>
  </form>
</div>
```

**"Clear all" placement:** `ms-auto` on the `<a>` pushes it to the far right within the flex row (D-UI2-10). Already exists as the last child; only the `ms-auto` positioning matters.

The `.overview-filter-bar` div no longer carries the `.panel` class (structural fix D-UI2-07).

**Picker macro:** `{% from "browse/_picker_popover.html" import picker_popover %}` — byte-stable, no fork (D-UI2-09).

## Pagination Contract (D-UI2-13, D-UI2-14)

### Placement Decision (Researcher's Call)

Pagination control belongs **in the sticky footer** alongside the entry count.

Rationale: The footer is the natural "summary bar" for the listing — it shows "N entries" (count) and the page navigation together. This mirrors the pattern of status bars in data tools and avoids cluttering the panel-body below the grid. The panel grid body scrolls vertically (`.overview-grid-body { max-height: 70vh; overflow-y: auto }`) — placing pagination at the panel bottom would require scrolling to reach it on longer lists, undermining its purpose.

Footer layout for JV page:
```
[  ← prev ]  [ 1 ] [ 2 ] [ 3 ] …  [ next → ]        6 entries
← pagination (Bootstrap .pagination)              count (ms-auto) →
```

### Server-Side Pagination (Researcher's Call)

Server-side pagination. Rationale: the existing HTMX `HX-Push-Url` URL-round-trip pattern (used for sort state) is cleanest with server-side slicing. The URL `?page=2&sort=customer&order=asc` fully encodes state. Client-side pagination would require JS state management outside HTMX's idiom.

### Constants and URL Shape

| Constant | Value | Location |
|----------|-------|----------|
| `JV_PAGE_SIZE` | 15 | `app_v2/routers/overview.py` constant (or `app_v2/services/pagination.py`) |
| URL query param | `page` | `/overview?page=2&sort=customer&order=asc` |
| Default page | 1 | Absent `page` param → page 1 |
| OOB block id | `overview-pagination` | `{% block pagination_oob %}` in overview/index.html |

### View Model Shape

Router injects into template context:
```python
{
  "rows": [...],          # current page rows only (slice of all_rows)
  "page": int,            # current page number (1-indexed)
  "page_count": int,      # ceil(total_count / JV_PAGE_SIZE)
  "total_count": int,     # total rows matching current filters
}
```

### Pagination OOB Block in overview/index.html

```html

        <a class="page-link"

           hx-push-url="true"
           aria-label="Previous">
          <i class="bi bi-chevron-left"></i>
        </a>
      </li>
      {# page number links — rendered by template loop or Jinja macro #}
      {# ellipsis logic: show first, last, and 2 pages around current; ellipsis otherwise #}
      <li class="page-item {% if vm.page >= vm.page_count %}disabled{% endif %}">
        <a class="page-link"
           hx-post="/overview/grid"
           hx-include="#overview-filter-form"
```

Wait — the count caption `#overview-count` is in the panel-header (D-UI2-11), not the footer. The footer carries the pagination control. The footer also carries the entry count on Browse. For JV, the panel-header already has the count (right side of panel-header). The footer carries pagination only.

```

```

The `#overview-count` OOB target stays in the panel-header. The `#overview-pagination` OOB target is in the footer.

### Ellipsis Logic for Page Numbers

Show up to 7 page items with ellipsis:
- Always show page 1 and page N (last).
- Always show current page ± 1.
- Insert `<li class="page-item disabled"><span class="page-link">…</span></li>` when gap > 1.

Implementation: Jinja macro in overview/index.html or a server-side helper that builds a `page_links` list of `(label, page_num | None)` tuples (None = ellipsis). Planner decides implementation detail.

### Filter + Sort + Page URL Round-Trip

All three dimensions round-trip via HX-Push-Url on every POST /overview/grid:
- `?sort=customer&order=asc&page=2&status=Pass&customer=Acme`
- Filter change → page resets to 1 (hidden `<input name="page" value="1">` in filter form).
- Sort change → page resets to 1 (sortable_th macro emits `hx-vals='{"page": "1", ...}'`).
- Page click → sort and filters preserved via `hx-include="#overview-filter-form"`.
