```python
def _execute_and_wrap(
    ctx: RunContext[ChatAgentDeps],
    sql: str,
    *,
    prefix_rejection: bool = True,
) -> str:
    """Verbatim port of nl_agent.run_sql — preserves SAFE-02..06 invariants.

    SAFE-02: validate_sql (single SELECT, no UNION/CTE/comments, allowed tables)
    SAFE-03: inject_limit (row cap)
    SAFE-04: SET SESSION TRANSACTION READ ONLY
    SAFE-04b: SET SESSION max_execution_time (timeout_s * 1000 ms)
    SAFE-06: scrub_paths (OpenAI only — D-CHAT-11)
    SAFE-05: <db_data>...</db_data> wrapper

    prefix_rejection=True: validator failures return 'REJECTED: <reason>' (D-CHAT-02);
    REPLACES the v1.0 'SQL rejected:' prefix so the chat-loop wrapper can string-prefix-match.
    """
    cfg = ctx.deps.agent_cfg
    vr = validate_sql(sql, cfg.allowed_tables)
    if not vr.ok:
        return f"REJECTED: {vr.reason}" if prefix_rejection else f"SQL rejected: {vr.reason}"

    safe_sql = inject_limit(sql, cfg.row_cap)
    timeout_ms = int(cfg.timeout_s) * 1000

    engine_fn = getattr(ctx.deps.db, "_get_engine", None)
    try:
        if engine_fn is None:
            df = ctx.deps.db.run_query(safe_sql)
        else:
            with engine_fn().connect() as conn:
                try:
                    conn.execute(sa.text("SET SESSION TRANSACTION READ ONLY"))
                except Exception:
                    pass
                try:
                    conn.execute(sa.text(f"SET SESSION max_execution_time={timeout_ms}"))
                except Exception:
                    pass
                df = pd.read_sql_query(sa.text(safe_sql), conn)
    except Exception as exc:
        return f"SQL execution error: {type(exc).__name__}"

    if df.empty:
        rows_text = "(no rows returned)"
    else:
        header = " | ".join(str(c) for c in df.columns)
        rows = "\n".join(" | ".join(str(v) for v in row) for row in df.itertuples(index=False))
        rows_text = f"{header}\n{rows}"

    if ctx.deps.active_llm_type == "openai":
        rows_text = scrub_paths(rows_text)
    return f"<db_data>\n{rows_text}\n</db_data>"
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| LLM → tool → DB | The agent's `run_sql` (and 4 other SQL-emitting tools) accepts LLM-generated SQL and executes it against MySQL; this is the primary attack surface for prompt-injection-driven SQL injection. |
| Tool result → LLM context | Tool return strings re-enter the LLM context window via PydanticAI's `message_history` replay; sensitive paths in result text could leak across turn boundaries. |
| Module-private `_execute_and_wrap` | Module-level helper used by all 5 SQL-emitting tools; a bug here regresses ALL tools simultaneously. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-02-01 | Tampering | LLM-generated SQL passed to `run_sql` | mitigate | PRIMARY backstop is the read-only DB user (CLAUDE.md project rule — even arbitrary SQL cannot write). SECONDARY backstop is `validate_sql` rejecting UNION/CTE/comments/non-SELECT/multi-statement; `inject_limit` capping rows; `SET SESSION TRANSACTION READ ONLY` defense-in-depth. Verbatim port from `nl_agent.run_sql` (already passing all SAFE-02..06 invariant tests). |
| T-03-02-02 | Tampering | LLM-generated SQL fragments passed to `count_rows`/`sample_rows` (where_clause appended into SELECT) | mitigate | The full assembled SQL passes back through `validate_sql` in `_execute_and_wrap`, so a malicious where_clause containing `UNION SELECT …` is rejected with `"REJECTED: UNION / INTERSECT / EXCEPT are not allowed"` — same backstop as run_sql. |
| T-03-02-03 | Information Disclosure | Filesystem-style strings (e.g. `/sys/block/...`) appearing in tool result text and being shipped to OpenAI | mitigate | `scrub_paths(rows_text)` applied when `ctx.deps.active_llm_type == "openai"` — verbatim port of SAFE-06. D-CHAT-11 carries this policy forward. (Plan 03 extends this to also scrub message_history before replay — that's the "both" decision from Claude's Discretion, applied at session-store boundary.) |
| T-03-02-04 | Information Disclosure | `<db_data>...</db_data>` wrapper around tool results | mitigate | The wrapper marks raw DB content so the LLM treats it as untrusted data, not instructions. The `_CHAT_SYSTEM_PROMPT` "CRITICAL SECURITY" block reinforces this verbally. Existing project pattern from nl_agent.py (SAFE-05). |
| T-03-02-05 | Denial of Service | Long-running SQL exhausting agent_cfg.timeout_s | mitigate | `SET SESSION max_execution_time={timeout_s * 1000}` enforces server-side cap; agent_cfg.timeout_s is bounded `[5, 300]`. Existing harness invariant. |
| T-03-02-06 | Denial of Service | Unbounded result rows OOMing the FastAPI process | mitigate | `inject_limit(sql, agent_cfg.row_cap)` injects/clamps LIMIT to row_cap (default 200, max 10_000). |
