) Click behavior (D-02, D-03): clicking a chip fills the textarea AND auto-submits via form.requestSubmit() so the question runs immediately. Inline onclick is acceptable per CONTEXT.md Claude's Discretion — no module-level JS needed for this trivial fill-and-submit operation. IMPORTANT — why data-question instead of {{ tojson }} in onclick: tojson produces a double-quoted JSON string (e.g. "My question"). Embedding that inside an HTML double-quoted attribute (onclick="...") terminates the attribute at the first embedded quote, producing broken HTML. The onclick handler would be silently truncated and the chip click would do nothing. Fix: store the question text in a data-question attribute (HTML-escaped, safe for attribute context) and read it via this.dataset.question in the onclick. No JS string literal appears inside the HTML attribute at all. The chips are NOT inside the #answer-zone — they live in their own container so an HTMX swap into #answer-zone doesn't replace them. Visibility lifecycle is governed by the parent index.html's {% if starter_prompts %} guard at server-render time (single-session-life). #}
